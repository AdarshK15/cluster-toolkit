📖 Context & Background for PR #5827
To help you understand exactly what you are reviewing, here is the full architectural context of how Slurm works on Google Cloud, the problem this PR solves, and why it matters.

1. Background: Slurm on GCP
Slurm is a highly scalable workload manager and job scheduler used heavily for High Performance Computing (HPC) and AI/ML workloads. In a Slurm cluster, there is a Controller Node (running the slurmctld daemon). The controller is the "brain" of the cluster—it manages the job queue, monitors compute nodes, and schedules tasks. If the controller goes down, the entire cluster halts; no new jobs can be scheduled, and running jobs may lose communication.

2. The Problem: Controller Failures in High Availability (HA)
To prevent the cluster from halting if the primary controller crashes, administrators configure High Availability (HA). This involves running a Primary controller and a Backup (Secondary) controller.

On Google Cloud, the Cluster Toolkit deploys these backup controllers inside Managed Instance Groups (MIGs). However, prior to this PR, if a backup controller VM froze, crashed, or became unresponsive (but the VM itself didn't terminate), Google Cloud didn't inherently know that the Slurm application inside the VM was broken. The controller would just sit there in a "zombie" state, requiring a human administrator to manually delete and recreate the VM.

3. The Solution: The "Hybrid" Approach & Auto-healing
According to Google's internal design documents (go/slurm-ha-ctk-design), the recommended approach to fix this is to combine Slurm's native HA capabilities with Google Cloud MIG Autohealing.

Autohealing works by:

Probing: Google Cloud constantly sends pings (Health Checks) to a specific port on the VM.
Detecting: If the Slurm controller stops responding to those pings for a certain amount of time, GCP marks the VM as "Unhealthy".
Healing: The MIG automatically deletes the unhealthy "zombie" VM and spins up a fresh, healthy replacement without any human intervention.
🔍 What this PR Actually Does
This Pull Request (PR #5827) implements the Autohealing solution for the Slurm HA controllers. It modifies the Terraform module (schedmd-slurm-gcp-v6-controller) to automatically wire up this resilience.

Specifically, it adds:

The Health Check (google_compute_health_check): A GCP resource that defines how to check if the controller is alive (e.g., pinging TCP port 6818 every 30 seconds). It allows users to customize the timeout, interval, and thresholds.
The Firewall Rule (google_compute_firewall): Google Cloud's health check probers originate from specific internal IP addresses (130.211.0.0/22 and 35.191.0.0/16). This PR automatically creates a firewall rule to allow those IPs to reach the Slurm controller.
Attaching to the MIG (auto_healing_policies): It attaches the newly created Health Check to the actual Managed Instance Groups (both Zonal and Regional variants) that house the backup controllers.
