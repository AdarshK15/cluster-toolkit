Here is the full context of what this Pull Request is trying to achieve, structured so you can clearly understand the feature you are reviewing.

🎯 High-Level Goal
This PR aims to introduce Private Service Connect (PSC) interface support to Slurm clusters provisioned by the Cluster Toolkit. PSC allows services to securely communicate across VPC networks without exposing traffic to the public internet.

🏗️ How it is Implemented
To support PSC, Google Cloud requires users to specify a network_attachment when configuring a virtual machine's network interface.

This PR accomplishes this by:

Expanding the Network Schema: It updates the additional_networks variable across all major Slurm modules (login nodes, controllers, nodesets, partitions, and internal instance templates) to accept a new network_attachment string.
Plumbing the Variable: It pipes this new network_attachment variable all the way down to the lowest-level Terraform resource (google_compute_instance and google_compute_instance_template).
Adding Validation: Because GCP requires that you use either standard networking (VPC/Subnet) or a PSC Network Attachment—but never both on the same interface—the author added Terraform validation blocks to ensure users don't configure conflicting inputs.
🔀 Why GKE is Excluded (The "Cross-Orchestrator" Aspect)
The Cluster Toolkit supports two main orchestrators: Slurm (using standard Compute Engine VMs) and GKE (Google Kubernetes Engine).

Currently, GKE's underlying networking schema (additional_node_network) does not natively support mapping PSC network_attachment interfaces in the same way standard VMs do. The PR author recognized this and intentionally isolated the feature so it only applies to Slurm modules. They also updated the internal Golang testing suite (pkg/inspect/modules_test.go) to test Slurm against the new PSC schema (wantPSC), while ensuring GKE continues testing against the legacy standard schema (want).

Safety / Guardrails: Are the Terraform validation rules robust enough to stop users from accidentally passing both a subnetwork and a network_attachment at the same time? (This is what the review feedback focuses on tightening up!)
